# テニス記録アプリ - Design

## プロダクト概要

**「テニスの家計簿」** - ざっくり記録→上達が見える

社会人テニスプレーヤー（週1〜2回）向けに、試合や練習をサクッと記録し、上達を可視化するアプリ。テニスベア（コート予約・マッチングアプリ）が「テニスする前」を解決しているのに対し、本アプリは「テニスした後」の振り返りを担う。

**ターゲット**: テニスベア等で仲間を見つけて試合している「楽しくやってるけど上手くなりたい」カジュアル上昇志向の社会人。競技選手・コーチ・学生部活は対象外。

### 解決すべきバーニングニーズ

| # | ニーズ | 具体的な困りごと |
|---|-------|----------------|
| 1 | **自分、上手くなってるのか問題** | 週1-2回プレーしても上達実感がない。レッスン代のROIが見えない |
| 2 | **試合の振り返りが記憶頼み問題** | 草トーや練習試合の結果を記録していない。「あの人に前も負けた気がする」レベルの曖昧さ。負けパターンを言語化できない |
| 3 | **対戦相手・仲間の情報管理問題** | テニスベアで知り合った人のレベル感を忘れる。「あの人とダブルス組むと相性いい」を感覚でしか持てない |

### 競合との棲み分け

```
テニスベア          DataTennis / SwingVision       本アプリ
(テニスする前)  →   (試合中の詳細記録)         →   (試合後のサクッと振り返り)
 予約・仲間集め       ガチ勢向け・入力が重い          ← ここが空いている
```

既存アプリ（DataTennis: ポイント単位スコア記録、SwingVision: 動画AI解析 年$180）はいずれもガチ勢向け。カジュアル層が15-20秒で記録できるアプリは存在しない。

---

## 設計原則

1. **入力は最小限、出力で感動させる** - 記録は15-20秒。価値はインサイトで返す
2. **相手情報は任意の付加データ** - 記録のコアループは相手データなしで成立させる（テニスベア経由で初対面の相手と複数試合するのが日常のため、毎回の相手登録は現実的でない）
3. **タイプ別に入力を最適化** - 練習/シングルス/ダブルスでMotivationが違う。入力量を合わせる
4. **データが溜まるほど価値が増す** - Hookedモデル（Trigger→Action→Variable Reward→Investment）で習慣化を設計する

## MVP機能スコープ（v0.1）

### P0（必須）

#### クイック記録（タイプ別）

最初の1タップでアクティビティタイプを選択し、タイプに応じた入力フォームに分岐する。

```
今日は何をした？
  🏋️ 練習・レッスン
  🎾 シングルス
  🎾🎾 ダブルス
```

**練習を記録対象に含める理由**: ターゲットユーザーの多くは毎週レッスンに通っている。試合だけでなく練習も記録することで、「ボレー練習を集中的にやった月 → 試合でのネットプレーが改善」のような因果が見えるようになり、バーニングニーズ1（上達実感・レッスン代ROI）を解決する。

タイプ別の入力項目:

| 入力項目 | 練習・レッスン | シングルス | ダブルス |
|---------|-------------|-----------|---------|
| スコア | - | ○ | ○ |
| 今日の調子（1タップ） | ○ | ○ | ○ |
| 相手（任意） | - | 任意 | - |
| パートナー（任意） | - | - | 任意 |
| 取り組んだこと（タグ） | ○ | - | - |
| メモ（任意） | ○ | ○ | ○ |

**「今日の調子」の自己評価**（😤ダメだった / 😐いつも通り / 😊良かった）は全タイプ共通の必須項目。1タップで完了し、時系列で溜まると「主観的な調子の推移」＝**上達実感の主軸**になる。相手データに依存しないため、どの記録パターンでも機能する。

**ダブルスで「対戦相手」ではなく「パートナー」を記録する理由**: バーニングニーズ3の「あの人とダブルス組むと相性いい」は、敵の情報より味方の情報の方が価値が高い。パートナーとの勝率を見ることで、最適な組み合わせが見つかる。

#### 対戦履歴タイムライン

- カレンダー表示で「どれくらいテニスしたか」可視化（試合・練習を色分け）
- 自動集計: 月間試合数、勝率、練習回数
- 自己評価（😤😐😊）の推移グラフ

#### Head-to-Head（相手を入れた試合のみ）

- 特定の相手/パートナーとの通算成績・直近傾向
- 「初対戦時は0-6, 0-6 → 最新は4-6, 5-7 → 接戦になってきた」のように改善が可視化される
- 相手データを入れた試合が2件以上で自動表示

Head-to-Headはバーニングニーズ1（特定の相手に対する上達の実感）とニーズ2（「あの人に前も負けた気がする」の解消）を同時に解決する。追加入力コストはゼロ（既存の相手データから集計するだけ）。

#### 記録後インサイト表示

記録完了直後に、蓄積データから自動生成したインサイトを1つ表示する。毎回異なる内容を出すことで、記録行為自体に「次は何が出るか」の楽しみを付与する（Hookedモデルの Variable Reward に該当）。

```
例:
- "初記録おめでとう! テニス記録の第一歩 🎾"
- "今月6試合目。先月の4試合をもう超えました"
- "😊 が3試合連続! 好調持続中"
- "田中さんに通算 3勝2敗! 勝ち越しています"
- "山田さんとのダブルス 4勝1敗! 最強コンビ"
```

データ蓄積量に応じて出せるインサイトの種類が増える（後述のHookedモデル参照）。

#### 対戦相手 / パートナー プロフィール（任意）

相手・パートナーの情報管理。登録のハードルを極限まで下げる設計:

- **名前だけで登録可能**（5秒）。レベルやプレースタイルは初回では聞かない
- 2回目以降は候補リストから選ぶだけ
- **レベルは2回以上対戦した時点でアプリから提案**:

```
"田中さんとは 1勝2敗。相手のレベルはどのくらい？"
  [自分より上] [同じくらい] [自分より下]
```

主観3段階なので正確さの負担がない。複数回対戦した常連相手だけにレベルが付くため、データの質も高い。

### P1（初期リリース後）

#### 月次サマリー

- 今月の試合数、勝率、練習回数、よく組んだパートナー、よく対戦した相手
- 先月との比較で変化を可視化

#### スキルタグ（試合記録への追加）

試合記録時にオプションで、良かった点・課題をタグ選択:

```
良かった: [サーブ] [ストローク] [ボレー] [メンタル]
課題:    [サーブ] [ストローク] [ボレー] [メンタル]
```

タップ式で0-2個選ぶだけ。P0で記録習慣が定着した後に、振り返りの解像度を段階的に上げる狙い。

#### マイルストーン通知

データから自動検出してプッシュ通知。記録のない日でもアプリを思い出すきっかけを作る:

```
- "🏆 格上に初勝利!"
- "🔥 3連勝中!"
- "📈 今月の勝率が先月より20%アップ"
- "🎾 通算50試合達成!"
```

### P2（検証後）

#### 詳細スキル評価

- 試合後に5項目を5段階評価（サーブ、リターン、ストローク、ネットプレー、メンタル）
- 時系列で推移を可視化
- P1のスキルタグで記録習慣がある前提の上位機能

#### 練習→試合の因果分析

- 「ボレー練習を5回やった月 → ネットプレーの😊が増加」のような因果を提示
- 練習記録と試合記録の両方を蓄積しているからこそ出せるインサイト
- バーニングニーズ1の「レッスン代のROI」に直接応える

## 記録フロー

### 練習・レッスン（15秒）

```
① 何をやった？（タグ選択、複数可）
   [サーブ] [ストローク] [ボレー] [フットワーク] [試合形式]
② 調子: 😤 😐 😊
③ メモ（任意）「スライスサーブのトスを前に」

→ 記録完了! + インサイト表示
```

スコアがないぶん最も軽い記録フロー。練習は試合より記録モチベーションが低いため、入力コストを最小にする。

### シングルス（20秒）

```
① スコア（6-4, 6-3 のドラムロール UI）
② 調子: 😤 😐 😊
③ 相手（任意: 過去の相手リストから選択 or 名前だけ新規追加）
④ メモ（任意）

→ 記録完了! + インサイト表示
```

### ダブルス（20秒）

```
① スコア（6-4, 6-3 のドラムロール UI）
② 調子: 😤 😐 😊
③ パートナー（任意: 過去のリストから選択 or 名前だけ新規追加）
④ メモ（任意）

→ 記録完了! + インサイト表示
```

### バッチ記録（イベント・草トー向け）

テニスベアのイベントや草トーナメントでは1日に3-6試合を異なる相手とこなすことが多い。通常フローで1試合ずつ記録するのは面倒なため、まとめて入力できるバッチモードを用意する。

```
"今日のまとめ記録"
  試合1: [勝ち ▼] [6-4, 6-2]
  試合2: [負け ▼] [3-6, 4-6]
  試合3: [勝ち ▼] [6-3, 6-1]
  今日の調子: 😊
  → 3試合を一括記録!
```

相手の名前は全部空欄でOK。「誰と」より「何試合やって、どう感じたか」を優先する。

## Hookedモデル設計

Nir Eyal の Hooked モデル（Trigger→Action→Variable Reward→Investment）に沿って、記録の習慣化を設計する。

### Trigger（きっかけ）

**External Trigger（初期〜習慣化まで）:**
- テニスをやりそうな曜日・時間帯にプッシュ通知: 「テニスお疲れ様! 30秒で記録 🎾」
- 週次サマリー完成時の通知: 「今週の振り返りができました」

**Internal Trigger（習慣化後）:**
- 勝った → 「記録して成長を残したい!」（達成感の保存欲）
- 負けた → 「何が悪かったか振り返りたい」（敗因の整理欲）
- ふと気になる → 「田中さんに何連敗だっけ...」（好奇心からH2Hを開く）

External Trigger は Internal Trigger が育つまでの補助輪。記録→インサイト表示の体験を数回繰り返すと、テニス後に自然とアプリを開くようになることを目指す。

### Action（行動）

15-20秒で完了する記録行為。**ユーザーのMotivationに応じて入力量が自然に増減する**設計:

| 状況 | Motivation | 入力パターン |
|------|-----------|------------|
| 試合に勝った | 高い | スコア + 調子 + 相手名 + メモまで全部入れたくなる |
| 試合に負けた | やや低い | スコア + 調子だけ。メモは気が向けば |
| 練習した | 最も低い | タグ + 調子だけ。スコアがないぶん最速で終わる |

最低限の入力（スコアor タグ + 調子の2タップ）でも記録として成立するため、Motivationが低い日でも脱落しにくい。

### Variable Reward（予測できない報酬）

記録完了直後に毎回異なるインサイトを1つ表示する。Nir Eyal の定義する3タイプの報酬を混ぜることで飽きさせない:

| 報酬タイプ | 例 | 効く場面 |
|-----------|---|---------|
| **Tribe**（社会的報酬） | "田中さんに勝ち越し!" "山田さんとのペア5勝1敗!" | 相手/パートナーを入力した時 |
| **Hunt**（情報的報酬） | "先月より勝率+20%" "ダブルスの方が勝率高い" | データが溜まってきた時 |
| **Self**（自己達成報酬） | "通算50試合!" "😊が3連続!" | いつでも出せる |

**「何が出るかわからない」から記録完了ボタンを押す瞬間に小さなワクワクが生まれる。**

データ量に応じて出せる報酬が段階的に増える:

| 蓄積データ量 | 出せる報酬の例 |
|------------|--------------|
| 1試合目 | 「初記録! テニス記録の第一歩 🎾」 |
| 3試合目 | 「3試合記録! 😊が2回、好調ですね」 |
| 同じ相手2回目 | 「田中さんとの2回目! 前回は4-6でした」 |
| 10試合 | 「10試合達成! シングルス勝率60%」 |
| 1ヶ月経過 | 「先月との比較が出ました!」 |
| 練習+試合蓄積 | 「サーブ練習の月は試合の調子が良い傾向」 |

初期でも必ず何かしらの報酬を出せることが重要。データが少ない段階では「N試合達成!」系のSelf報酬を出し、蓄積に応じてHunt/Tribe報酬を追加していく。

### Investment（投資）

記録するたびにアプリの価値が不可逆的に上がる構造。ユーザーが「ここまで溜めたデータを捨てたくない」と感じることが、長期定着の鍵:

- **データ蓄積** → インサイトの質・種類が増える（報酬が豊かになる）
- **H2Hデータ充実** → 特定の相手との「対戦の物語」が生まれる
- **パートナー相性** → ダブルスの最適な組み合わせが見つかる
- **練習×試合の因果** → レッスン代のROIが可視化される（P2）
- **長期記録** → 「自分のテニス人生の記録」として替えがきかない個人資産になる

Investment が次の Trigger を強化するサイクル: データが溜まる → より面白いインサイトが出る → 次も記録したくなる。

## 意図的にスコープ外とするもの

| 機能 | 外す理由 |
|------|---------|
| 動画録画・AI解析 | SwingVision（年$180のAI分析アプリ）の領域。開発コスト大、差別化困難 |
| ポイント単位のスコア記録 | DataTennis（ポイント単位記録アプリ）の領域。入力が重すぎてカジュアル層は続かない |
| コート予約・マッチング | テニスベア（予約・マッチングアプリ）の領域。補完であって競合しない |
| SNS/コミュニティ機能 | 初期はシンプルに個人ツールとして価値を証明する |

## 技術スタック

### 選定方針

**MVP（v0.1〜0.2）**: localStorage で完結。サーバー不要。
**v0.3〜**: Supabase 導入で友達機能・データ同期を追加。

移行コストを最小化するため、データアクセスをカスタムhooks（`useActivities`, `usePersons`等）で抽象化し、localStorage → Supabase の差し替えをUI無変更で行えるようにする。

### ホスティング: exe.dev

exe.dev は SSH 接続で使える軽量 VPS サービス。

- `ssh exe.dev` でVM作成、sudo権限あり、Ubuntu環境
- HTTPS + SSL証明書が自動付与される
- デフォルトprivate、共有リンクで公開可能（ベータテスト配布に便利）
- Docker対応

**MVPのデプロイ方法**: VM上で `vite build` → `npx serve dist/` で静的配信。Vercel等の外部サービスは不要。

### フレームワーク・ライブラリ

| カテゴリ | 選定 | 理由 |
|---------|------|------|
| ビルド | **Vite + React 19** | ビルド成果物を exe.dev 上で静的配信するだけ。シンプル |
| スタイリング | **Tailwind CSS 4** | モックで既に使用。ユーティリティファーストで高速開発 |
| UIコンポーネント | **shadcn/ui** | コピペ方式でバンドルサイズ最小。Drawer, Dialog, Tabs等が揃っている |
| 状態管理 | **Zustand + persist middleware** | localStorage永続化が組み込み。軽量（1KB）。後でSupabase同期に差し替え可能 |
| ルーティング | **React Router v7** | SPA で十分。ページ遷移は少ない（タブ3つ + モーダル） |
| 日付操作 | **date-fns** | 月間集計・カレンダー表示に必要。day.jsより tree-shakeable |
| ID生成 | **nanoid** | オフライン環境でユニークID生成。後のSupabase移行時にUUID変換可能 |
| PWA | **vite-plugin-pwa** | ホーム画面追加でネイティブ風体験。オフライン動作保証 |

### なぜ Next.js ではなく Vite か

- MVP は完全クライアントサイド（API なし、SSR なし）
- exe.dev は VM なので、ビルド済み静的ファイルを配信するだけで十分
- v0.3 で Supabase 入れる際も Vite + React のまま対応可能
- Next.js の App Router は今回オーバースペック

### v0.3 移行時の変更点

| 項目 | MVP（localStorage） | v0.3（Supabase） |
|------|---------------------|-------------------|
| データ層 | Zustand + persist | Zustand + Supabase realtime sync |
| 認証 | なし | Supabase Auth（Google / Apple） |
| API | なし | Supabase PostgREST（自動生成） |
| ホスティング | exe.dev（静的配信） | exe.dev（同じ） |

## データモデル

### エンティティ

```typescript
// 記録タイプ
type ActivityType = 'singles' | 'doubles' | 'practice'

// 自己評価（全記録タイプ共通、1タップで入力）
type SelfRating = 'bad' | 'neutral' | 'good'  // 😤 😐 😊

// 相手レベル（2回以上対戦した相手にのみ、アプリ提案で設定）
type RelativeLevel = 'above' | 'same' | 'below'

// 練習で取り組んだ内容のタグ
type PracticeTag = 'serve' | 'stroke' | 'volley' | 'footwork' | 'match-play'

// 対戦相手 / パートナー（共通エンティティ）
interface Person {
  id: string           // nanoid
  name: string         // 必須（これだけで登録可能）
  relativeLevel?: RelativeLevel  // 2回目以降にアプリが提案
  note?: string
  createdAt: string    // ISO 8601
  updatedAt: string
}

// 試合記録（シングルス / ダブルス）
interface Match {
  id: string
  type: 'singles' | 'doubles'
  date: string         // YYYY-MM-DD
  sets: Set[]          // [{myScore: 6, opponentScore: 4}, ...]
  result: 'win' | 'lose'  // sets から自動算出
  selfRating: SelfRating
  opponentId?: string  // シングルス時の相手 Person.id（任意）
  partnerId?: string   // ダブルス時の味方 Person.id（任意）
  memo?: string
  createdAt: string
  updatedAt: string
}

interface Set {
  myScore: number      // 0-7
  opponentScore: number // 0-7
}

// 練習記録
interface Practice {
  id: string
  type: 'practice'
  date: string
  tags: PracticeTag[]  // 取り組んだ内容（複数選択）
  selfRating: SelfRating
  memo?: string
  createdAt: string
  updatedAt: string
}

// 全記録の Union 型
type Activity = Match | Practice

// ユーザープロフィール（ローカル保存）
interface UserProfile {
  nickname: string
  experience?: '< 1 year' | '1-3 years' | '3-5 years' | '5+ years'
  onboardingCompleted: boolean
  createdAt: string
}
```

### Zustand Store 構成

```
stores/
├── useActivityStore.ts    // Match + Practice の CRUD + 集計ロジック
├── usePersonStore.ts      // Person CRUD（相手・パートナー共通）
├── useProfileStore.ts     // ユーザー設定 + オンボーディング状態
└── useInsightStore.ts     // インサイト生成ロジック
```

各 store は `persist` middleware で localStorage に自動保存。

### v0.3 Supabase 移行時のテーブルマッピング

```sql
profiles (id, nickname, experience, ...)
persons (id, user_id, name, relative_level, note, ...)
activities (id, user_id, type, date, self_rating, memo, result, ...)
```

## バーニングニーズの充足マッピング

| バーニングニーズ | P0での対応 | 追加入力コスト |
