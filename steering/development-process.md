# AI駆動開発プロセス設計

このドキュメントはプロジェクトを **AIエージェントチームで並列開発する** ための開発プロセスを定義する。

## 設計原則

### コンテキストは有限。増えるほどアホになる

LLMの根本的な制約:
- コンテキストウィンドウには上限がある
- コンテキストが増えると注意力が分散し、出力品質が劣化する
- 1つのエージェントに全部やらせるとコンテキストが膨張してパフォーマンスが落ちる

**対策**: 専門エージェントに分割し、各エージェントには **最小限のコンテキスト** だけを渡す。

### ファイルをインターフェースにする

エージェント間の連携は **会話ではなくファイルの入出力** で行う:
- 各タスクは TaskCreate の description に具体的に記述
- エージェントの出力はファイル（コード修正）に残る
- これにより各フェーズが疎結合になり、途中からのやり直しや並列実行が容易になる

### AIが主導者、人間は最小限の作業だけ

**原則**: 人間に確認を求めて手を止めるのは最終手段。判断に迷ったら「後で変えやすい方」を選んで進む。

## Claude Code の3層構造

### Layer 1: Skills（知識の外部化）

- `implement-feature`: React コンポーネント実装手順
- `review-code`: design-direction.md との照合レビュー手順

### Layer 2: Custom Subagents（専門ワーカー）

- `feature-implementer`: タスク駆動の機能実装（worktree隔離可）
- `code-reviewer`: デザイン方針との照合レビュー（読み取り専用）
- `build-tester`: 型チェック + ビルド確認
- `code-patrol`: 品質パトロール（読み取り専用）

### Layer 3: Agent Teams（並列チーム）

設定: `.claude/settings.local.json` で `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`

---

## コンテキスト管理のベストプラクティス

### 1. エージェントには最小限のコンテキストだけ渡す

```
Good: "OnboardingPage.tsx L112 の navigate('/record/singles') を RecordTypeSheet 表示に変更してください"
Bad:  "design-direction.md を読んでオンボーディングを修正してください"
```

### 2. 失敗したら該当エージェントだけやり直す

ビルドエラーが出たら、該当ファイルの担当だけ再実行する。
