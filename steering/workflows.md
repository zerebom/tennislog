# TennisLog - ユーザーワークフロー集

MVP以降のフェーズを見据えた、主要ワークフローの整理。

---

## WF-1: オンボーディング（初回起動）

### ゴール
30秒以内にアプリの価値を体感させ、最初の1記録を完了させる

### フロー

```
① アプリ起動
   └→ スプラッシュ（TennisLog ロゴ、0.5秒）

② ウェルカム画面（1枚だけ）
   ┌─────────────────────────────┐
   │  🎾 試合を記録して、        │
   │     上達を見える化しよう     │
   │                             │
   │  ・30秒で記録              │
   │  ・勝率が見える            │
   │  ・対戦相手の情報を管理     │
   │                             │
   │  [ はじめる ]               │
   └─────────────────────────────┘
   ※ チュートリアルスライドは作らない（離脱の元）

③ プロフィール設定（最小限）
   - ニックネーム（必須）
   - テニス歴（任意、ドロップダウン: 1年未満/1-3年/3-5年/5年以上）
   ※ 写真・メール等は後回し

④ 「最初の記録をしよう」への誘導
   ┌─────────────────────────────┐
   │  最近テニスしましたか？      │
   │                             │
   │  [ 記録する ]  ← 主導線     │
   │  [ あとで ] ← ホームへ      │
   └─────────────────────────────┘
   ※「あとで」を選んでもホームで空状態メッセージ
     「最初の記録をして、あなたのテニスログを始めましょう」

⑤ タイプ選択 → 記録フロー（WF-2/WF-2b/WF-3）

⑥ 完了 → ホームに1件目のカードが表示される
   + インサイト: 「初記録おめでとう! テニス記録の第一歩 🎾」
```

### 設計判断
- **アカウント登録は不要**（ローカル保存 → ソーシャル機能追加時にアカウント化）
- **チュートリアルは作らない**（UIが十分シンプルなら不要）
- **最初の記録を「体験させる」のがKPI**（オンボーディング完了率 = 初回記録完了率）

## WF-2: 試合記録（シングルス）

### ゴール
20秒以内にシングルスの試合結果を記録し、インサイトを表示する

### フロー

```
＋ボタン（FAB）
         ↓
タイプ選択:
┌──────────────────────────────────────┐
│  今日は何をした？                     │
│                                      │
│  🏋️ 練習・レッスン                   │
│  🎾 シングルス        ← これを選択   │
│  🎾🎾 ダブルス                       │
└──────────────────────────────────────┘
         ↓
1画面フォーム:
┌─────────────────────────────┐
│ シングルスを記録             │
│                              │
│ スコア                       │
│ [6 ▼]-[4 ▼]  [6 ▼]-[3 ▼]  │
│ ← ドラムロール式             │
│                              │
│ 今日の調子                   │
│ [😤] [😐] [😊]  ← 1タップ  │
│                              │
│ 相手（任意）                 │
│ [田中さん] [鈴木さん] [+新規]│
│ ※ 空欄でもOK                │
│                              │
│ メモ（任意）                 │
│ [________________]           │
│                              │
│ [ 記録する ]                 │
└─────────────────────────────┘
         ↓
記録完了 + インサイト表示（WF-7）
```

### 設計判断
- **スコアはドラムロール**（キーボード入力より速い）
- **相手は任意**（入れなくても記録として成立する）
- **resultはsetsから自動算出**（ユーザーに勝ち/負けを選ばせない）

│                              │

│                              │

## WF-3: 練習記録

### ゴール
15秒以内に練習内容を記録する。スコアなし、タグと調子だけ。

### フロー

```
＋ボタン → タイプ選択で「🏋️ 練習・レッスン」
         ↓
1画面フォーム:
┌─────────────────────────────┐
│ 練習を記録                   │
│                              │
│ 何に取り組んだ？（複数可）   │
│ [サーブ] [ストローク]        │
│ [ボレー] [フットワーク]      │
│ [試合形式]                   │
│                              │
│ 今日の調子                   │
│ [😤] [😐] [😊]              │
│                              │
│ メモ（任意）                 │
│ [________________]           │
│ 例:「スライスサーブのトスを前に」│
│                              │
│ [ 記録する ]                 │
└─────────────────────────────┘
         ↓
記録完了 + インサイト表示（WF-7）
```

### ホームでの表示

```
直近のアクティビティ（試合+練習 統合タイムライン）

🎾 vs 田中さん      6-4, 6-3  😊    3/1
🏋️ サーブ,ストローク          😐    2/28
🎾🎾 w/ 山田さん   6-3, 6-1  😊    2/28
🏋️ ボレー,フットワーク       😤    2/27
```

### 設計判断
- **スコアなし、時間なし**（練習のMotivationは最も低い。入力は最小限に）
- **タグ選択で取り組み内容を構造化**（後でP2の因果分析に使える）
- **調子（😤😐😊）は共通**（練習でも自己評価の推移が追える）

## WF-3b: バッチ記録（テニスベアイベント向け）

### ゴール
1日に複数試合やった場合に、まとめて記録できる

### フロー

```
＋ボタン → 「今日のまとめ記録」（バッチモード）
         ↓
┌─────────────────────────────┐
│ 今日のまとめ記録             │
│                              │
│ 試合1:                       │
│ [勝ち ▼] [6-4, 6-2]         │
│                              │
│ 試合2:                       │
│ [負け ▼] [3-6, 4-6]         │
│                              │
│ 試合3:                       │
│ [勝ち ▼] [6-3, 6-1]         │
│                              │
│ [+ 試合を追加]               │
│                              │
│ 今日の調子（全体）           │
│ [😤] [😐] [😊]              │
│                              │
│ [ 3試合を一括記録 ]          │
└─────────────────────────────┘
         ↓
記録完了 + インサイト表示
```

### 設計判断
- **相手の名前は全部空欄でOK**（「誰と」より「何試合やって、どう感じたか」が先）
- **調子は1日単位で1つ**（試合ごとに聞くと入力が重くなる）
- **テニスベアのイベント帰りにサクッと記録するユースケースを想定**

## WF-4: 仲間の登録・編集

### 新規登録フロー（記録フロー内で発生）

```
記録フォームで [+新規] タップ
         ↓
ボトムシートが開く:
┌─────────────────────────────┐
│ 新しい仲間を追加             │
│                              │
│ 名前 *                       │
│ [________________]           │
│                              │
│ メモ（任意）                 │
│ [________________]           │
│                              │
│ [ 追加 ]                     │
└─────────────────────────────┘
         ↓
自動で相手/パートナーチップが選択される → 記録続行
```

### レベル提案（2回目以降の対戦時にアプリが自動提案）

```
同じ相手と2回以上対戦した時点で表示:
┌─────────────────────────────┐
│ 田中さんとは 1勝2敗。       │
│ 相手のレベルはどのくらい？   │
│                              │
│ [自分より上] [同じくらい]    │
│ [自分より下]                 │
└─────────────────────────────┘
※ 主観3段階。NTRPのような難しい判断は不要
※ スキップ可能
```

### 後から編集（仲間詳細画面から）

```
仲間タブ → カードタップ → /persons/:id
         ↓
ページ上部に [編集] ボタン
         ↓
同じボトムシートが開く（既存情報がプリフィル）
```

### 設計判断
- **必須項目は名前だけ**（5秒で登録完了）
- **レベルは初回登録時には聞かない**（入力負荷を最小化）
- **レベルは主観3段階（自分より上/同じ/下）**（4段階レベルやプレースタイルは廃止）
- **記録フローの中で自然に登録**（わざわざ仲間タブに行く必要なし）
- **「対戦相手」→「仲間」に統一**（相手もパートナーも同じPersonモデル）

## WF-5: 友達機能（ソーシャル）

### フェーズ分け

| Phase | 機能 | 解放条件 |
|-------|------|---------|
| Phase 1（MVP） | ローカルの仲間管理 | - |
| Phase 2 | 友達申請・承認 | ユーザー数500+ |
| Phase 3 | 友達のタイムライン閲覧 | Phase 2稼働後 |
| Phase 4 | 対戦結果の共有・相互確認 | Phase 3稼働後 |

### Phase 2: 友達申請

```
仲間タブ → 相手カードの [...] メニュー
         ↓
┌──────────────────────┐
│ 友達として招待        │  ← アプリ内ユーザーを検索
│ QRコードで追加        │  ← オフラインで出会った相手
│ リンクをシェア        │  ← LINE/メッセンジャー経由
└──────────────────────┘

【QRコード方式（テニスコートで使う想定）】
┌─────────────────────────────┐
│ 📱 あなたのQRコード         │
│                              │
│       ┌─────────┐           │
│       │ QR CODE │           │
│       └─────────┘           │
│                              │
│ 相手にこのQRを読み取って    │
│ もらうと、友達になれます     │
│                              │
│ [ スキャンする ]             │
└─────────────────────────────┘

承認フロー:
相手に通知 → 承認/拒否 → 承認されたら相互に友達リストに追加
```

### Phase 3: 友達タイムライン

```
ホームタブにセグメント追加:
┌──────────┬──────────┐
│  自分     │ フレンド │
└──────────┴──────────┘

【フレンドタイムライン】
┌─────────────────────────────┐
│ 田中さんが試合を記録しました │
│ 🎾 vs 山田さん 6-3, 6-4     │
│ 3/1 有明テニスの森           │
│ [👍 いいね] [コメント]       │
│─────────────────────────────│
│ 鈴木さんが練習を記録しました │
│ 🏋️ サーブ,ストローク        │
│ 2/28                         │
│ [👍 いいね]                  │
└─────────────────────────────┘
```

### Phase 4: 対戦結果の相互確認

```
自分が田中さんとの試合を記録
         ↓
田中さんに通知: 「Hさんが試合結果を記録しました。確認しますか？」
         ↓
田中さん側: [確認する] → スコアを見て [OK] or [修正リクエスト]
         ↓
双方が確認済み → 記録に ✅ マークが付く
```

### 設計判断
- **Phase 1はソーシャル不要**（個人ツールとしてまず価値を証明）
- **QRコードは必須**（テニスコートで出会った人とすぐ繋がれる）
- **公開範囲は友達のみ**（全体公開しない = プライバシー重視）
- **いいね/コメントは軽量に**（SNS化しすぎない。モチベーション維持が目的）
- **対戦結果の相互確認は信頼性担保**（「盛った」と思われない）

## WF-7: 記録後インサイト表示

### ゴール
記録完了直後に、データから自動生成したインサイトを1つ表示。Variable Rewardとして記録継続のモチベーションにする。

### フロー

```
記録完了
         ↓
┌─────────────────────────────┐
│  ✅ 記録しました!            │
│                              │
│  💡 インサイト               │
│  ─────────────────────────  │
│  "今月6試合目。               │
│   先月の4試合をもう超えました"│
│                              │
│  [ ホームへ ]                │
└─────────────────────────────┘
```

### データ蓄積量に応じたインサイトの例

| 蓄積データ量 | インサイト例 | タイプ |
|------------|-------------|-------|
| 1試合目 | 「初記録おめでとう! テニス記録の第一歩 🎾」 | Self |
| 3試合目 | 「3試合記録! 😊が2回、好調ですね」 | Self |
| 同じ相手2回目 | 「田中さんとの2回目! 前回は4-6でした」 | Tribe |
| 10試合 | 「10試合達成! シングルス勝率60%」 | Self |
| 同じ相手3回目+ | 「田中さんに通算3勝2敗! 勝ち越しています」 | Tribe |
| ダブルス相性 | 「山田さんとのダブルス 4勝1敗! 最強コンビ」 | Tribe |
| 1ヶ月後 | 「先月との比較が出ました!」 | Hunt |
| 連続好調 | 「😊が3試合連続! 好調持続中」 | Self |
| 練習+試合蓄積 | 「サーブ練習を増やした月は試合の調子が良い傾向」 | Hunt |

### 設計判断
- **毎回1つだけ表示**（多いと読まれない）
- **データがないうちはSelf系**（達成マイルストーン中心）
- **データが溜まるとTribe/Hunt系が増える**（社会的・分析的な報酬）
- **Hookedモデルの「Variable Reward」に対応**
